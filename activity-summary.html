Parsing engine: iteration history

Build

v1 — flat-text heuristic parser

First pass parsed ML Kit's flattened text string line-by-line, matching a trailing price on each line.

Bug found

Column bleed on real thermal receipt

Tested against a real coffee-shop receipt photo (Armor Genuine, 11 items). ML Kit groups OCR text into spatial blocks, not visual rows — the flattened text interleaved the name column and the price column out of order. "Total Item 12" was even misread as the grand total (value 12).

Fix

v2 — bounding-box row reconstruction, then nearest-neighbor pairing

Switched to using each recognized line's bounding box. First tried bucketing lines into rows by vertical overlap — broke on the photo's slight skew. Replaced with direct nearest-neighbor pairing: every price is matched to the single closest label that structurally looks like a label ("1x 28.000", "Total", "Tax"), only falling back to "closest text of any kind" when no such label exists. Also normalized OCR's "1x" → "lx" (digit 1 / letter L) confusion.

"1x 28.000 OREO CI..." · 34000
After
"OREO CRUSHER" · 34000
Bug found

Digit misreads in individual prices

Names/pairing were now correct, but 4 of 11 prices had OCR digit noise, e.g. ARTOMORO GULA AREN showed 28080 instead of 28000.

Fix

v3 — cross-validate the two OCR reads of the same price

Every "1x 28.000"-style label already contains a second, independent OCR read of the unit price, previously discarded. Now both readings are compared and whichever is a clean round-hundred value is preferred (retail prices on this receipt type are never odd amounts).

Bug found

Smoothlies Mango still wrong (32 instead of 32000)

The one remaining case: both OCR reads were corrupted — the amount column dropped a digit ("32.00", read as a decimal instead of thousands), and the label had a wrong trailing digit ("32.009").

Fix

v4 — prefer the correct magnitude, then round

Added a tie-breaker: when neither reading is clean, prefer whichever raw string has the proper "thousands" digit-group shape (a wrong trailing digit is far less damaging than a dropped digit that corrupts the whole magnitude), then snap to the nearest hundred.

Bug found

Second receipt: completely different layout

A table-style invoice (Butik Liceria) with proper columns — KETERANGAN / JUMLAH / HARGA / TOTAL — plus new OCR letter/digit confusions (Rpl20.000,00, Rpó0.000,00). None of it matched the thermal-receipt heuristics.

Fix

v5 — table-layout detection + dedicated table parser

Parser now detects a header row containing 2+ recognizable column keywords (English or Indonesian) and, when found, clusters lines into rows and assigns each cell to its nearest column by position — falling back to the original thermal-receipt logic when no table header is detected. Also added a normalization pass for letter-for-digit OCR confusions inside amounts.
